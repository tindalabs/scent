# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

This is the curated, repo-wide view. Per-package changelogs (generated by Changesets) live alongside each package: [`scent-sdk`](packages/sdk/CHANGELOG.md), [`scent-engine`](packages/engine/CHANGELOG.md), [`scent-otel`](packages/otel/CHANGELOG.md).

## [Unreleased]

Post-0.1.0 hardening and dependency modernization. These changes touch the dev
tree, CI, the (private) server, and the Observatory only — there's no
consumer-visible change to the published packages, so no version bump is
warranted yet; they roll into the next release.

### Fixed
- **`risk_assessments.flags` stored double-encoded**: the insert used `${JSON.stringify(flags)}::jsonb`, which postgres.js double-encodes into a JSON *string* scalar instead of an array. `POST /v1/resolve` then spread that string into per-character garbage (a long array of empty `{}` flags) and produced a `NaN`→`null` risk score. Switched the insert to `sql.json(flags)` and hardened the resolve route to ignore non-array stored flags. Forward-only: the project is unreleased, so no deployed database carried the bad data, and the defensive read plus self-healing on the next assessment cover any stale local-dev rows. (Found via end-to-end demo testing.)
- **Automation-detector wording**: the anti-tamper flag now reads "Anti-tamper signals" (not "Automation detected") when the combined confidence is weak — e.g. devtools open in a dev environment — so a human isn't labelled a bot. The machine-readable `code` (`automation_suspected`) is unchanged; the reason text also drops the `tamper.` prefix for readability.

### Internal
- **Admin two-factor auth (TOTP)** (migration 011): authenticator-app codes (otplib) with hashed one-time recovery codes; the shared secret is AES-256-GCM-encrypted at rest via a new `SCENT_SECRET_KEY` app key (unset = 2FA enrollment disabled, server still runs). Login gains a second-factor step (code or recovery code). Owners can require 2FA install-wide; un-enrolled admins are funneled into setup (login still issues a session, so no lockout). Final PR of the 3-PR admin arc. New env var `SCENT_SECRET_KEY` (`openssl rand -hex 32`) — documented in the deploy runbook + image config.
- **Admin account management** (Observatory Users page + migration 010): owner-only invite via copy-paste link (opaque single-use token, 7-day expiry, no SMTP) with a public `/accept-invite` flow that auto-logs-in; list/role-change/soft-deactivate; per-project membership grants; self-service password change. Deactivation (`admin_users.is_active`) revokes sessions immediately and blocks login; lockout guards prevent demoting/deactivating yourself or the last active owner. Second of the 3-PR admin arc (TOTP 2FA next).
- **Admin RBAC (two-level)**: added `admin_users.role` (owner/member) and a `project_members(user_id, project_id, role)` table (migration 009). Owners are superusers; members reach only granted projects, with a per-project role (`admin` = manage keys, `viewer` = read-only). Enforced in `requireProjectRead` (data reads) and the `/admin/projects` routes (create/delete owner-only; rotate requires owner or project-admin; list scoped per user). CLI-created admins are owners; existing accounts were migrated to owner. First of a 3-PR arc (account-management UI and TOTP 2FA to follow).
- **Per-project data viewing in the Observatory**: the `/v1` read routes now accept an admin session + `X-Project-Id` header (GET only) alongside the project API key (`requireProjectRead`), and the Observatory gained a sidebar project switcher that scopes every data page to the selected project. The build-time `VITE_API_KEY` is removed — one admin login can browse any project's data. Ingest (`/v1/events`) and synchronous resolve stay strictly key-gated; a session can never reach a write path.
- **Structured logging** ([#19]): replaced ad-hoc `console.*` in the server with a `pino` logger (JSON to stdout, level via `LOG_LEVEL`). Imported after the OTel SDK starts so its pino instrumentation injects `trace_id`/`span_id` into log lines.
- **SDK test coverage** ([#20]): the persistence layers (cookie / localStorage / sessionStorage / IndexedDB via `fake-indexeddb`, incl. cross-layer "storage amnesia" resurrection) and previously-untested collectors, with privacy invariants (WebRTC opt-in, no raw UA, no PII).
- **Server integration tests** ([#21]): `events.ts` identity resolution driven end-to-end with supertest, behind a new `createApp()` factory (`index.ts` is now a thin bootstrap). CI gained **PostgreSQL + Redis service containers**; the suite is gated on `DATABASE_URL` (runs in CI, skips locally without a DB).
- **Resolution-algorithm edge cases** ([#23]): confidence/continuity band cutoffs, weighted-Jaccard (absent/empty/moderate), the SimHash candidate threshold, drift classification branches, and the orchestration decisions — new-vs-returning boundary, ambiguous match, and cluster linking.

## [0.1.0] - 2026-06-05

First public release. Published to npm as [`@tindalabs/scent-sdk`](https://www.npmjs.com/package/@tindalabs/scent-sdk), [`@tindalabs/scent-engine`](https://www.npmjs.com/package/@tindalabs/scent-engine), and [`@tindalabs/scent-otel`](https://www.npmjs.com/package/@tindalabs/scent-otel), with signed provenance via npm Trusted Publishing. (`@tindalabs/scent-server` is private and unpublished.)

### Added
- **SDK** — `init(options)` + `observe()` for browser signal collection and identity resolution; `flush()` to send buffered snapshots; configurable `persistence` modes (`conservative` / `balanced` (default) / `aggressive` / `forensic`).
- **18 signal collectors** — canvas, audio, fonts, screen, locale, hardware, platform, touch, network, plugins, media, storage-mode, and anti-tamper (WebDriver, headless, patched-API, DevTools, entropy-spoof). WebRTC is opt-in.
- **Probabilistic engine** (`scent-engine`) — SimHash candidate retrieval + drift-tolerant weighted-Jaccard similarity, confidence/continuity bands, and drift classification with per-signal entropy.
- **Risk engine** — detectors for automation, entropy instability, storage amnesia, rapid re-registration, impossible transition, coordinated behavior, and `coordinated_accounts`; composite 0–1 score with risk band.
- **Account linking & entity graph** — `scent.identify(accountId)` links a device identity to an application account; `POST /v1/identity/:id/link`, `GET /v1/identity/:id/accounts`, and `GET /v1/account/:accountId/identities` (the fraud-detection query); a `GET /v1/accounts/clusters` aggregate; the `coordinated_accounts` flag (≥3 distinct accounts within a 30-day window); and an Observatory "Linked accounts" panel + "Account clusters" view.
- **Server** — REST API (`/v1/events`, `/v1/identity`, `/v1/identities`, `/v1/resolve`, `/v1/clusters`, `/v1/account`, `/v1/accounts`, `/v1/dashboard`), PostgreSQL with versioned migrations, Redis-backed rate limiting and API-key caching.
- **OpenTelemetry** (`scent-otel`) — `readTraceparent`, `attachScentAttributes`, and `ScentOtelBridge`; server spans for identity resolution, risk assessment, and account linking.
- **Observatory** — dashboard, identity list/detail, drift timeline, and cluster views.
- **Infra** — CI (lint / type-check / build / test / audit), CodeQL, per-package READMEs + badges, grouped Dependabot, and a Trusted-Publishing release workflow.
